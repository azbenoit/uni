A pseudorandom number generator (PRNG) is an algorithm producing a sequence that behaves similarly to a sequence of random numbers, but is generated deterministically starting from some initial seed. What “behaves similarly to” means can depend on context, and different applications require PRNGs approximating different properties of true randomness, such as being uniformly distributed, or being difficult to predict without knowing the initial seed.

In this section, you will implement a simple kind of PRNG known as a linear feedback shift register (LFSR). To explain how this works, let’s consider the following illustration of an LFSR generating 16-bit numbers:

Huffman coding

Lossless data compression is the operation of transforming a sequence of bits into a shorter sequence of bits that encodes the same information – the initial sequence of bits being reobtained using a decompression algorithm. In short, it is a coding operation which shortens the size (transmission, storage) of the data at the cost of compression work.

Huffman coding is a variable length coding, developed by David Albert Huffman while a student at MIT. The algorithm has been published in 1952 in a paper entitled “A Method for the Construction of Minimum-Redundancy Codes”.

In this section, we are interested in compressing (character) strings. In a normal mode of operation, when using the extended ASCII encoding, strings are encoded using 8 bits per character. To get the binary representation of a string, we can use the .encode() method. For example, the following function takes a string, gets its ASCII encoding (i.e. as an array of bytes that is represented using the class bytearray in Python) and formats it so that it is human readable (i.e. returns a string made of 0s and 1s, one per bit, where groups of 8 bits are separated by a space - again, for readability).

def bin_repr_of_ascii_string(s):
    return ' '.join(format(x, '08b') for x in s.encode('ascii'))

For instance, on the string "abc", we obtain the following representation:

01100001 01100010 01100011

i.e., the ASCII encoding of "abc" is 3 bytes long (as expected) and one can check, for example, that the byte 0b01100001 (i.e. 97 in the decimal notation) corresponds to the character a in the ASCII table.

(Note that in extended ASCII, since characters are encoded using 8-bit words, you are limited to 256 characters. Basically, the encoding space is barely large enough to encode latin characters (without diacritic marks, a.k.a. accents). As of today, there exist dozens of different encodings, some of them being multi-byte (i.e. more than one byte is used for encoding a single character) or even variable-width (i.e. the encoding of one character may vary in size). For instance, Python uses internally UTF-16 (a 2-byte long encoding) for storing characters. For this tutorial, we will stick to extended ASCII.)

In Huffman, compression is achieved by replacing the most frequent characters with short codes (< 8 bits) and the least frequent characters with long codes (> 8 bits). In average, size reductions of between 20 and 90% are thus observed. It is a widely used compression method, often as a complement to other algorithms.

Huffman uses a particular method to choose the representation of each coded symbol. The idea is that each code should not be a prefix of any other code. In other words, none of the codes chosen should be the beginning of another code. We will see that more in details later.

Description of the Huffman coding

The principle of Huffman coding is based on the creation of a tree structure called an Huffman tree. That tree is built as follows: suppose the sentence ss to be compressed is

Forsan et haec olim meminisse juvabit

We first compute the probability of appearance, in the sentence, of each character. For example, in ss, the character c appears once, while the character a and the space character appear 33 and 55 times, respectively. Since ss is of length 3737, the inverse probabilities of appearance of c, a and the space character are resp. equal to 137137, 337337 and 537537. We give below the probability of appearance for all the characters of the sentence:

Then, the tree structure is constructed as follows. For each character cc with probability pp, we start by constructing a set of isolated leaf nodes (i.e. without any child) with data cc and associated probability pp. Then, we iteratively repeat the following process: we select two nodes with lowest associated probabilities and remove them from the set – let’s call them n1n1 and n2n2 and assume that they have associated probabilities p1p1 and p2p2. We then create a new node with children n1n1 and n2n2 (the order is irrelevant here), with no data and with associated probability p1+p2p1+p2, and add it to the set. Eventually, the set will only contain one node that forms the Huffman tree.

To obtain the Huffman code of a character cc, we walk down the tree from the root to the leaf that contains cc, adding a 0 (resp. a 1) to the code each time we take a left-edge (resp. a right-edge).

The construction of an Huffman tree for the probability table given above is shown below:

If we want, e.g., to find the code for u, it suffices to walk the tree from the root down to the node u (the curly path in the picture above), adding a 0 (resp. a 1) to the code each time we take a black left-edge (resp. a red right-edge). In that case, we obtain the code 10010. You can check that the letters with a higher probability of appearance have shorter codes.

Last, one can encode an entire string by encoding its characters one by one, using their respective Huffman codes, and by concatenating all the obtained codes.

Binary Huffman Implementation

In the previous section, we saw the principle of Huffman encoding and developed basic encoding & decoding functionalities. However, to obtain a functional implementation, some work remains:

  • first, for representing bit-strings, we used (character) strings made of 0s and 1s. That is, we used 1 character (i.e. 8 bits if we assume the ASCII encoding of characters) for encoding 1 bit of information! In that setting, the function huffman_encode is more an anti-data-compression implementation than a data-compression one.

  • then, the functions huffman_encode & huffman_decode assume that you have the relevant Huffman tree at disposition. For huffman_encode, this is not a problem since you have the non-encoded message at disposition and the tree can be reconstructed from it. But for the huffman_decode function, you cannot assume that – the goal of the function is to decode the message without already having access to its decoding! In consequence, the Huffman tree must be part of the encoded message, i.e. we have to find a way to encode an Huffman tree using bit-strings. Note that encoding must be efficient (in term of space) because now, the encoding of the Huffman tree is going to be part of compressed data – the more space it takes, the less the compression ratio will be. (There are other solutions. E.g., the Huffman tree could be fixed a priori and be a pre-shared knowledge between the compressor and the decompressor. For example, fax machines1 use a modified Huffman algorithm where the Huffman tree is fixed for all machines & transmissions.)

  • last, for strings that are made from only one character (e.g. aaaaaa), the Huffman tree is made of a single node. In consequence, the repeated character is going to be encoded with a code of length 00. And thus, all strings of that form are going to be compressed as the empty bit-string and uncompressed as the empty string – you can check this with your implementation. We will have to detect such cases.

Ok, we are not far from the end. What remains is to (binary) encode the Huffman tree and to pack all things together. The are numerous ways of encoding an Huffman tree. I here give one among them:

  • the bit encoding of a leaf node is the bit 1 followed by the 8-bit long ASCII code of the character this node encodes (as returned by ord),

  • the bit encoding of an internal node is the bit 0 followed by the encoding of its left children and its right children (in that order).

Finally, the binary encoding of an Huffman tree is simply the binary encoding of its root padded with 0s at the end if the encoding length (in bits) is not a multiple of 8.
